# PR 규칙

## 기본 원칙
- 모든 기능/버그 수정은 PR을 통해서만 `develop` 에 머지
- `main` 직접 push 금지 — `develop` → `main` PR로만 머지
- PR은 하나의 기능 단위로 작게 유지

## PR 제목
```
[feat] 로그인 JWT 발급 구현
[fix] 재고 차감 동시성 오류 수정
[docs] API 명세 업데이트
[chore] Docker Compose 설정 추가
```

## PR 본문 템플릿
```markdown
## 작업 내용
- 변경한 내용 요약

## 변경 파일
- `AuthService.java` — 로그인 로직 구현
- `AuthController.java` — 로그인 API 엔드포인트

## 테스트
- [ ] 단위 테스트 통과
- [ ] 관련 API 동작 확인

## 참고
- 관련 이슈 또는 문서 링크
```

## 머지 전 체크리스트
- [ ] 브랜치명이 컨벤션에 맞는가
- [ ] 커밋 메시지가 컨벤션에 맞는가
- [ ] 테스트가 모두 통과하는가
- [ ] 민감 정보가 코드에 포함되지 않았는가
- [ ] 불필요한 주석/디버그 코드가 없는가
