# Week 4 리팩토링 이력 (2026-05-21)

> 대상: order-service · product-service  
> 브랜치: `feature/order-service`

## CRITICAL

### C-01 — Kafka 이벤트 발행 시점 보장
- **문제**: `@Transactional` 내부에서 Kafka 직접 발행 시 DB 롤백 후 이벤트가 이미 발행된 상태 발생 가능
- **해결**: `ApplicationEventPublisher` + `@TransactionalEventListener(AFTER_COMMIT)` 패턴 도입
  - `OrderService` → `applicationEventPublisher.publishEvent(new OrderCreatedApplicationEvent(...))`
  - `OrderKafkaEventRelay` → `AFTER_COMMIT` 시점에 Kafka 발행
- **파일**: `order-service/event/OrderCreatedApplicationEvent.java` (신규), `OrderKafkaEventRelay.java` (신규), `OrderService.java` (수정)

### C-02 — `@Transactional` self-invocation 우회
- **문제**: `StockDecreaseService.decreaseStock()` 내부에서 같은 Bean의 `@Transactional` 메서드 호출 → AOP 프록시 미적용
- **해결**: `StockDecreaseTransactionService` 별도 Bean으로 분리 → 프록시를 통한 호출 보장
- **파일**: `product-service/service/StockDecreaseTransactionService.java` (신규), `StockDecreaseService.java` (수정)

### C-03 — Kafka 발행 결과 비동기 에러 처리
- **문제**: `kafkaTemplate.send()` 결과를 무시 → 발행 실패 시 로그 없음
- **해결**: `whenComplete()` 핸들러로 성공/실패 로깅
- **파일**: `product-service/event/StockEventPublisher.java`

## HIGH

### H-02 — 멱등성 키 설정 시점
- **문제**: 재고 차감 전 멱등성 키 설정 → JVM 크래시 시 처리되지 않은 주문 영구 skip
- **해결**: 재고 차감 성공 확정 후 키 설정 (`opsForValue().set()`)
- **파일**: `product-service/service/StockDecreaseService.java`

## MEDIUM

### M-04 — 외부 API 호출 범위
- **문제**: `productClient.getProduct()` 가 `@Transactional` 내에 포함 → 불필요하게 트랜잭션 유지
- **해결**: `createOrder()` (비트랜잭션) → `fetchOrderItems()` → `saveOrderAndPublishEvent()` (트랜잭션) 구조 분리
- **파일**: `order-service/service/OrderService.java`
