# MSA 서비스 간 보안 — X-User-Role 헤더 위조 방지

> 면접 대비 정리 문서

---

## 현재 구조

```
클라이언트
    ↓
Gateway: JWT 검증 → X-User-Role: ADMIN 헤더 주입
    ↓
Product Service: 헤더만 보고 ADMIN 체크
```

Gateway를 통해서만 들어온다는 걸 암묵적으로 신뢰하는 구조.

---

## 문제점

서비스 포트가 외부에 노출되면 헤더 위조로 우회 가능.

```
악의적 요청
POST http://localhost:8082/api/v1/products
X-User-Role: ADMIN   ← 직접 위조 → ADMIN API 접근
```

---

## 해결 방법

### 1. 내부망 격리 (기본, Week 6 적용 예정)

```yaml
# gateway만 외부 포트 노출
gateway:
  ports:
    - "8080:8080"

# 나머지 서비스 — ports 없음 (내부 네트워크만)
auth-service:
  networks:
    - ecommerce-net

product-service:
  networks:
    - ecommerce-net
```

외부에서 8081, 8082 직접 접근 불가 → 헤더 위조 불가.

### 2. 서비스 메시 — Istio (운영)

```
서비스 간 통신에 mTLS 자동 적용
인증서 없는 요청은 거부
K8s 환경에서 사용
```

### 3. 내부 JWT (대형 서비스)

```
Gateway가 서비스 전용 내부 토큰 발급
각 서비스가 내부 토큰 검증
카카오/네이버급에서 사용
```

---

## 실무 선택 기준

| 규모 | 선택 |
|------|------|
| 중소 서비스 | 내부망 격리만으로 충분 |
| 대형 서비스 | 서비스 메시 (Istio) |
| 카카오/네이버급 | 내부 JWT + 서비스 메시 |

---

## 면접 예상 질문

**Q. X-User-Role 헤더 위조를 어떻게 막나요?**
> Docker 네트워크로 서비스 포트를 외부에 노출하지 않아 Gateway를 통해서만 접근 가능하도록 합니다. 운영 환경에서는 Istio 서비스 메시로 mTLS를 적용해 인증서 없는 서비스 간 통신을 거부합니다.

**Q. 지금 구조가 제로트러스트인가요?**
> 아닙니다. Gateway를 암묵적으로 신뢰하는 구조입니다. 진짜 제로트러스트는 모든 서비스 간 통신에 mTLS를 적용하는 서비스 메시 수준이 필요합니다.

**Q. MSA에서 서비스 간 인증을 어떻게 처리하나요?**
> 기본적으로 내부망 격리로 외부 접근을 차단하고, Gateway가 JWT를 검증해서 사용자 정보를 헤더로 전달합니다. 더 강한 보안이 필요하면 서비스 메시(Istio)로 mTLS를 적용하거나 내부 전용 토큰을 사용합니다.
